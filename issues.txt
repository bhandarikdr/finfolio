Is there any logical ambiguity regarding the use and implementation of primary index? Is there any implementation hurdle?

From a technical and architectural standpoint, the implementation of the Primary Market Index is now quite robust, but there are a few subtle logical areas where "ambiguity" or "hurdles" could exist depending on your future needs.
Here is an analysis of the remaining logical considerations:
1. The "Identity Conflict" Ambiguity
The Hurdle: What happens if a user renames the Primary Index to something that already exists as a secondary index (e.g., renaming "NEPSE Index" to "Banking Sub-Index")?
•
Logical Ambiguity: The database uses indexName as a Primary Key. If the scraper finds real "Banking Sub-Index" data and tries to save it, and you also have a Primary Index named "Banking Sub-Index" that was mapped from NEPSE data, one will overwrite the other.
•
Current Solution: We handle this by making the Primary Index "fixed." In the configuration list, we hide or lock duplicates, but if names match exactly, the data might "flicker" between NEPSE values and Sector values depending on which source is scraped last.
2. Scraper "Strictness" vs. "Flexibility"
The Hurdle: Different websites use slightly different names for the core index (e.g., "NEPSE Index" vs. "Nepse" vs. "Nepal Stock Exchange").
•
Logical Ambiguity: Our current normalization logic is very strict (lower == "nepse index" || lower == "nepse"). If a new fallback source uses "NEPSE Price" or just "Index", the mapping to your custom name might fail.
•
Implementation Hurdle: To fix this, we would need a "Fuzzy Matching" engine or a list of "Core Aliases." For now, the "Not Found" warning label we added is the best defense because it alerts the user that the scraper isn't recognizing the source data.
3. Metric Attribution (Mapping Logic)
The Hurdle: You mentioned that "User defined index should be set... searched in urls... if no metrics found, default 0.00."
•
Logical Ambiguity: There is a distinction between Searching for the Custom Name and Mapping NEPSE to the Custom Name.
•
Current Strategy: The app assumes your Custom Name represents the core market index (NEPSE). It scrapes the core index and labels it with your name.
•
The Hurdle: If you want to track a completely different index that isn't NEPSE (e.g., "NIFTY 50" from an Indian source), the scraper logic would need to be updated to recognize different table structures entirely. Currently, the scrapers are optimized for Nepali market layouts.
4. Visibility Persistence Logic
The Implementation Hurdle: Handling the transition of names in a comma-separated string (visibleIndicesJson) in a singleton database row.
•
The Hurdle: When you rename an index, we have to surgically clean that string to prevent "Ghost" entries (indices that are checked but no longer exist).
•
Robustness: The current fix in PortfolioRepository handles this by converting the JSON/String to a MutableSet, removing the old name, and re-saving. This is the correct approach for Room/SQLite.
Summary
The biggest hurdle is Data Consistency. If the user sets a custom name, they are essentially creating a "Nickname" for the market's core performance. As long as the scraper can identify what "Core" is, the logic holds.
Is there any ambiguity left? The only remaining ambiguity is User Intent: Does the user want the custom name to replace NEPSE data, or to look for a new type of data?
•
